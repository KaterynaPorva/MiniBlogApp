# Веб-додаток "Міні-блог"

## Опис проєкту

**Назва проєкту:** Міні-блог  
**Мета:** Створення навчального веб-додатку, що моделює функціональність невеликої блог-платформи.

### Основний функціонал:
- Реєстрація та авторизація користувачів
- Створення, редагування та видалення власних постів
- Коментування постів інших користувачів
- Можливість ставити лайки
- Перегляд аналітики постів
- Перегляд журналу дій

## Технології
- **Мова програмування:** C#
- **Фреймворк:** ASP.NET Core (Razor Pages)
- **Зберігання даних:** In-Memory
- **Сесії:** ASP.NET Session (для авторизації користувача)

## 🧩 Використані патерни проєктування

### 🛠️ 1. Породжувальні патерни (Creational Patterns)

| Патерн | Де використано | Призначення |
|--------|---------------|------------|
| **Singleton** | BlogSettings, BlogStorage | Гарантує існування єдиного екземпляра налаштувань та сховища в пам'яті (In-Memory) протягом усього життєвого циклу програми |
| **Builder** | PostBuilder | Дозволяє покроково створювати складні об'єкти Post. Використовує Fluent Interface для зручного налаштування заголовка, автора та вмісту |
| **Factory Method** | LogFactory та спадкоємці | Інкапсулює логіку створення різних типів записів у лог (PostLog, LikeLog, CommentLog), дозволяючи розширювати систему новими типами подій без зміни існуючого коду |

---

### 🏗️ 2. Структурні патерни (Structural Patterns)

Це найбільш насичена частина проєкту, де реалізовано складну систему "обгорток".

#### 🧱 Facade (BlogFacade)

Виступає єдиною точкою входу для UI-шару (`IndexModel`, `ViewPost`).  
Приховує складну взаємодію між:
- сховищем  
- парсером Markdown  
- системою сповіщень  
- командами  

**Результат:**  
Контролери та сторінки стали максимально легкими (*Thin Controllers*).

---

#### 🔄 Adapter (MarkdigAdapter)

Адаптує зовнішню бібліотеку **Markdig** до внутрішнього інтерфейсу `IMarkdownParser`.

**Результат:**  
Ми не залежимо від конкретної бібліотеки. Якщо потрібно змінити Markdig — достатньо створити новий Adapter.

---

#### 🛡️ Proxy (CachedBlogStorageProxy)

Кешуючий замісник для сховища:
- якщо пост є в кеші — повертає його миттєво  
- якщо пост відсутній — звертається до базового сховища  
- після отримання — зберігає результат у кеш  

---

#### 🎨 Decorator (LoggingBlogStorageDecorator)

Додає функціонал логування до будь-якої реалізації `IBlogStorage` без зміни їхнього коду.

**Архітектурна особливість:**

---

#### 🌿 Composite (Comment)

Представляє ієрархічну структуру коментарів:
- кожен коментар може містити список інших коментарів (відповідей)  
- реалізовано деревоподібну структуру  

**Результат:**  
Нескінченна вкладеність обговорень з можливістю рекурсивного підрахунку через `GetTotalCount()`.

### 🧠 3. Поведінкові патерни (Behavioral Patterns)

| Патерн | Де використано | Призначення |
|--------|---------------|------------|
| **Strategy** | IPostSortStrategy | Дозволяє динамічно змінювати алгоритм сортування постів (за датою або за популярністю) |
| **State** | IPostState (Draft, Moderation, Published) | Змінює поведінку поста залежно від його статусу |
| **Command** | LikeCommand, CommandManager | Інкапсулює дію "Поставити лайк" та дозволяє реалізувати Undo (скасування) |
| **Observer** | IBlogObserver, BlogFacade | Реалізує систему підписок та сповіщень |
| **Template Method** | PostSortStrategy | Визначає скелет алгоритму сортування, залишаючи конкретну реалізацію підкласам |

## 💎 Дотримання принципів SOLID

- **S (Single Responsibility Principle)**  
  Кожен клас відповідає за одну задачу  
  *(наприклад, PostBuilder тільки будує об'єкти, а BlogFacade тільки координує роботу компонентів)*  

- **O (Open/Closed Principle)**  
  Система відкрита для розширення (нові стратегії, стани, логування), але закрита для змін існуючого коду  

- **L (Liskov Substitution Principle)**  
  Всі реалізації `IBlogStorage` (включно з Proxy та Decorator) можуть взаємозамінювати одна одну без помилок  

- **I (Interface Segregation Principle)**  
  Клієнти залежать від вузьких інтерфейсів (`IMarkdownParser`, `IActivityLogger`)  

- **D (Dependency Inversion Principle)**  
  Високорівневі модулі не залежать від низькорівневих.  
  Всі залежності впроваджуються через **Dependency Injection** у `Program.cs`

## Документація та CI/CD

Цей проєкт використовує **Doxygen** для автоматичної генерації технічної документації, яка охоплює всі моделі, сервіси, Razor Pages, утиліти та Unit-тести.

## Автоматизація (GitHub Actions)
Для генерації та публікації створено workflow `.github/workflows/docs.yml`, який реалізує повний цикл CI/CD:
1. **Встановлення:** Налаштовує середовище з Doxygen та Graphviz.
2. **Генерація:** Створює HTML-документацію на основі коментарів у коді.
3. **Деплой:** Автоматично публікує оновлену документацію на GitHub Pages.

Workflow запускається автоматично при push у гілку `feature/docs-ci` або вручну.

## Тестування (Unit Tests)

Для забезпечення стабільності роботи веб-додатку створено юніт-тести для основних компонентів.

**LoggerTests**
- **LoggerServiceAdvancedTests.cs** — перевірка логування дій користувачів (створення постів, лайків, коментарів).

**Сервісні тести (ServiceTests)**
- **BlogStorageAddPostTests.cs** — перевірка додавання нових постів.
- **BlogStorageAddLikeTests.cs** — перевірка додавання лайків, уникання дублікатів та поведінки при відсутності посту.
- **BlogStorageAdvancedTests.cs** — комплексне тестування додавання коментарів, лайків, постів та логування подій.
- **BlogStorageEdgeCaseTests.cs** — перевірка граничних випадків (дуже довгий контент, порожні коментарі, чутливість до регістру у іменах користувачів).
- **UserServiceTests.cs** — перевірка базової аутентифікації та отримання списку користувачів.
- **UserServiceAdvancedTests.cs** — додаткові перевірки на чутливість до регістру та обробку порожнього пароля.

**Утиліти (UtilsTests)**
- **PostAnalyzerTests.cs** — перевірка аналізу постів (кількість лайків та коментарів, обробка постів без автора або заголовку).
- **PostStatsUtilsTests.cs** — перевірка підрахунку сумарної статистики по всіх постах та по конкретному користувачу.

**Колекції тестів**
- **BlogStorageTestsCollection.cs** — вимкнення паралельного виконання тестів для BlogStorage.
